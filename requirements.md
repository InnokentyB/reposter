# Требования к боту репоста из Telegram в VK, Одноклассники и Threads

## 1. System model

### Цель системы

Нужно создать бота/сервис, который автоматически дублирует все новые посты из одного Telegram-канала в:

- сообщество VK
- группу или страницу в Одноклассниках
- аккаунт Threads

Система должна быть пригодна для постоянной эксплуатации: переживать временные ошибки API, не публиковать один и тот же пост многократно, давать понятный статус доставки и позволять безопасно менять настройки интеграций.

### Предполагаемая модель работы

- Telegram является источником истины для контента.
- На каждый новый пост в канале создается одна внутренняя задача публикации.
- Доставка в VK, Одноклассники и Threads выполняется независимо по каждому каналу назначения.
- Ошибка в одной платформе не должна блокировать публикацию в остальных.
- Для каждого назначения хранится отдельный статус: `pending`, `published`, `failed`, `retry_scheduled`, `manual_review_required`.

### Основные акторы

- Владелец бота: настраивает интеграции и наблюдает за состоянием системы.
- Telegram Bot API или Telegram Client API: источник событий или механизм чтения постов канала.
- API VK.
- API Одноклассников.
- API Threads или утвержденный способ публикации в Threads.
- Планировщик/воркер: отвечает за повторы, отложенные публикации и повторную обработку сбоев.

### Активы, которые нужно защищать

- токены Telegram, VK, Одноклассников, Threads
- идентификаторы каналов, групп и страниц
- контент постов до и после преобразования
- журнал публикаций и статусы доставки
- конфигурация маршрутов "источник -> назначения"
- логи и диагностические события

### Границы доверия

- Входящие события от Telegram нельзя считать полностью надежными без дедупликации.
- Любой внешний API может отвечать с задержкой, ошибкой, rate limit или частичным успехом.
- Контент Telegram-поста может содержать форматирование, ссылки, медиа и спецсимволы, которые не переносятся одинаково между платформами.
- Операционные действия администратора должны быть ограничены и аудируемы.

### Ограничения и допущения

- Предполагается один исходный Telegram-канал на первом этапе.
- Предполагается один набор назначений на старте: один VK, один OK, один Threads-аккаунт.
- Репост делается только для новых постов после включения системы, если отдельно не реализован backfill.
- Комментарии, реакции, статистика и синхронизация правок поста не входят в обязательный MVP, если не указано иное.
- Для Threads нужно отдельно подтвердить допустимый способ автоматической публикации, так как политика и доступность API могут меняться.

### Вероятные домены отказа

- Telegram доставил событие повторно или не по порядку.
- Платформа назначения приняла пост, но ответ потерялся.
- Медиа не подходит под ограничения конкретной платформы.
- Истекли или были отозваны токены.
- Система перезапустилась между получением события и завершением публикации.
- Накопилась очередь из-за rate limit или деградации одной из платформ.

## 2. Findings

### Finding 1

- Severity: critical
- Area: интеграция Threads
- Problem: способ автоматической публикации в Threads не зафиксирован как поддерживаемый и стабильный.
- Why it matters: можно спроектировать весь поток вокруг канала, который позже окажется недоступным, ограниченным или рискованным для аккаунта.
- Likely trigger or failure path: выбор неофициальной интеграции, изменение API/политики, блокировка токена или невозможность публиковать нужный тип контента.
- Exact fix or design correction: выделить Threads в отдельный адаптер с явным флагом `beta`, заранее зафиксировать поддерживаемый механизм публикации и предусмотреть возможность временно отключить только Threads без влияния на VK и OK.

### Finding 2

- Severity: critical
- Area: идемпотентность публикаций
- Problem: без устойчивого ключа дедупликации один Telegram-пост может быть опубликован повторно при ретраях, рестартах или повторной доставке события.
- Why it matters: дубли постов в соцсетях быстро приводят к репутационным потерям и ручной чистке.
- Likely trigger or failure path: воркер упал после успешной публикации, но до сохранения статуса; Telegram прислал событие повторно; оператор нажал "повторить" без защиты.
- Exact fix or design correction: присвоить каждому исходному посту глобальный `source_post_id` на основе `telegram_chat_id + telegram_message_id`, хранить отдельный уникальный ключ публикации на каждую платформу и запрещать повторную отправку без проверки уже сохраненного результата.

### Finding 3

- Severity: high
- Area: преобразование контента
- Problem: форматирование, ссылки, альбомы, видео, опросы и подписи не совместимы между платформами один в один.
- Why it matters: пост может частично потеряться, стать нечитаемым или быть отклонен API назначения.
- Likely trigger or failure path: Telegram Markdown/HTML не конвертируется корректно, превышены лимиты длины текста, альбом не поддерживается целевой платформой.
- Exact fix or design correction: добавить нормализованный внутренний формат поста и отдельные platform-specific renderers с проверкой ограничений по длине, типу медиа и допустимому форматированию.

### Finding 4

- Severity: high
- Area: частичные сбои и согласованность
- Problem: публикация в три платформы имеет независимые исходы, но без раздельного учета статусов система будет терять точную картину.
- Why it matters: невозможно понять, где пост уже опубликован, а где нужен повтор или ручное вмешательство.
- Likely trigger or failure path: VK успешно опубликовал, OK вернул 500, Threads завис по timeout, а система записала общий статус "ошибка".
- Exact fix or design correction: хранить статусы доставки по каждому destination отдельно, поддерживать отдельные retry policy и показывать агрегированный итог только как производное состояние.

### Finding 5

- Severity: high
- Area: секреты и доступы
- Problem: токены нескольких платформ и права на публикацию являются высокопривилегированными секретами.
- Why it matters: утечка даст злоумышленнику возможность публиковать от имени сообществ или читать служебные данные.
- Likely trigger or failure path: хранение в `.env` без ротации, попадание токенов в логи, общий админский токен на все операции.
- Exact fix or design correction: хранить секреты только в защищенном secret store или как минимум в изолированных переменных окружения с маскированием, разделить токены по платформам, исключить их из логов и предусмотреть штатную ротацию.

### Finding 6

- Severity: high
- Area: управление нагрузкой и rate limit
- Problem: внешние API соцсетей ограничивают частоту публикаций и медиа-операций.
- Why it matters: bursts из Telegram или массовый backfill могут привести к блокировке, длинным очередям и каскадным повторам.
- Likely trigger or failure path: несколько постов подряд, повторная обработка после простоя, бесконтрольные ретраи без backoff.
- Exact fix or design correction: использовать очередь задач, ограничение параллелизма по платформе, экспоненциальный backoff с jitter и ceiling на число повторов.

### Finding 7

- Severity: medium
- Area: наблюдаемость и поддержка
- Problem: без сквозной трассировки публикации сложно разбирать инциденты и ручные повторы.
- Why it matters: оператор не поймет, что именно пошло не так, и будет принимать решения вслепую.
- Likely trigger or failure path: жалоба "пост не ушел в OK", при этом в логах нет связки между исходным сообщением и попытками публикации.
- Exact fix or design correction: ввести correlation id на пост, структурированные логи, счетчики успехов/ошибок/ретраев по платформам и журнал операторских действий.

### Finding 8

- Severity: medium
- Area: изменение и удаление постов
- Problem: не определено, что делать при редактировании или удалении поста в Telegram после первоначальной публикации.
- Why it matters: каналы назначения могут остаться в несогласованном состоянии, а логика обновлений окажется опасной при неправильной идентификации постов.
- Likely trigger or failure path: автор исправил текст в Telegram, а система либо молча проигнорировала это, либо создала новый пост вместо обновления старого.
- Exact fix or design correction: для MVP явно зафиксировать политику: либо "репостим только первичную публикацию", либо "поддерживаем edit-sync только для текста", с отдельными тест-кейсами и возможностью выключить эту функцию.

## 3. Proposed changes

### Минимальная архитектура для надежного baseline

1. Источник событий:
   получать новые посты из Telegram и приводить их к внутреннему каноническому формату.
2. Хранилище:
   сохранять пост, его вложения, маршруты доставки и статусы публикации по платформам.
3. Очередь задач:
   на каждую платформу создавать отдельную задачу публикации.
4. Адаптеры платформ:
   `vk-publisher`, `ok-publisher`, `threads-publisher` с единым интерфейсом.
5. Воркер повторов:
   переотправляет только безопасные и идемпотентные задачи.
6. Панель или admin CLI:
   позволяет видеть статусы, отключать назначения и инициировать ручной retry.

### Функциональные требования

- Система должна отслеживать все новые посты заданного Telegram-канала.
- Система должна поддерживать как минимум текстовые посты, посты с одним изображением и посты с несколькими изображениями, если это допускается платформой назначения.
- Система должна сохранять исходный текст и нормализованную версию контента до начала публикации.
- Система должна публиковать в VK, OK и Threads независимо друг от друга.
- Система не должна повторно публиковать один и тот же Telegram-пост в одну и ту же платформу без явного ручного override.
- Система должна фиксировать идентификатор созданного поста на стороне платформы назначения.
- Система должна уметь помечать публикацию как `manual_review_required`, если автоматический retry исчерпан или контент несовместим с платформой.
- Система должна позволять временно выключить отдельную платформу без остановки всего конвейера.
- Система должна логировать причину отказа в человекочитаемом и машинно-обрабатываемом виде.

### Нефункциональные требования

- Идемпотентность:
  каждая публикация в конкретную платформу должна иметь уникальный ключ и безопасно переживать повторные запуски.
- Надежность:
  сбой одной платформы не должен блокировать доставку в остальные.
- Наблюдаемость:
  должны быть метрики по числу принятых постов, успешных публикаций, ошибок, ретраев и задач в очереди.
- Безопасность:
  токены и конфигурация интеграций не должны попадать в репозиторий, логи и сообщения об ошибках.
- Восстановление:
  после рестарта система должна продолжить обработку незавершенных задач без ручной чистки состояния.
- Расширяемость:
  добавление новой платформы должно требовать реализации нового адаптера, а не переписывания ядра.

### Требования к данным

Нужны как минимум следующие сущности:

- `source_posts`
  поля: `id`, `source_platform`, `source_channel_id`, `source_message_id`, `published_at`, `raw_payload`, `normalized_payload`, `content_hash`
- `destinations`
  поля: `id`, `platform`, `target_id`, `status`, `config_ref`
- `delivery_jobs`
  поля: `id`, `source_post_id`, `destination_id`, `status`, `attempt_count`, `next_attempt_at`, `last_error_code`, `last_error_message`
- `published_posts`
  поля: `id`, `delivery_job_id`, `remote_post_id`, `published_at`, `remote_permalink`
- `audit_events`
  поля: `id`, `actor`, `action`, `target_type`, `target_id`, `result`, `created_at`

### Требования к обработке ошибок

- На timeout внешнего API система должна делать ограниченный retry с backoff.
- На ошибки валидации контента система не должна бесконечно ретраить задачу.
- На rate limit система должна уважать рекомендованную задержку, если она передана API.
- При неясном исходе публикации система должна сначала попытаться определить, был ли пост создан, и только потом повторять отправку.
- При недоступности одной платформы очередь этой платформы должна замедляться независимо от остальных.

### Требования к безопасности

- Все входящие данные от Telegram должны валидироваться по схеме и ограничиваться по размеру.
- Все токены должны храниться вне кода и быть доступны сервису по принципу least privilege.
- Логи не должны содержать access token, refresh token, cookie, webhook secret и полный сырой ответ API, если в нем есть чувствительные данные.
- Доступ к административным операциям должен быть ограничен списком разрешенных операторов.
- Все ручные действия: retry, disable destination, rotate token, remap target должны писаться в audit log.

### Требования к эксплуатации

- Нужен health endpoint или эквивалентный механизм проверки состояния.
- Нужен dead-letter сценарий для задач, которые не удалось обработать автоматически.
- Нужен простой способ переобработать диапазон пропущенных постов без дублирования уже опубликованных.
- Нужен dashboard или хотя бы CLI-команда для просмотра зависших задач и последних ошибок.
- Нужен конфиг уровня окружения: `dev`, `staging`, `prod`, с изоляцией токенов и целей публикации.

### Решения для MVP

- На первом этапе поддерживать только один исходный Telegram-канал.
- На первом этапе не синхронизировать удаления из Telegram в платформы назначения.
- На первом этапе редактирование Telegram-поста либо игнорировать, либо обрабатывать только как отдельную явно включаемую функцию.
- Threads запускать за feature flag до подтверждения стабильности интеграции.

### Приоритет реализации

P0:

- прием новых постов из Telegram
- каноническая модель поста
- идемпотентная запись задач доставки
- публикация в VK и OK
- статусы по платформам
- безопасное хранение токенов
- retry/backoff/dead-letter
- базовые логи и метрики

P1:

- Threads-адаптер под feature flag
- ручной retry из admin CLI
- backfill пропущенных постов
- алерты по repeated failure

P2:

- частичная синхронизация edits
- шаблоны трансформации контента по платформам
- несколько исходных каналов

## 4. Residual risks and follow-ups

### Что остается неясным и требует подтверждения

- Какой именно тип Telegram-источника будет использоваться: bot webhook, polling или client API.
- Какие типы контента обязательны для MVP: текст, фото, видео, альбомы, документы, опросы, ссылки.
- Допустимы ли разные шаблоны текста для разных платформ.
- Нужна ли публикация "сразу" или допускается задержка в несколько минут.
- Какой инструмент публикации в Threads будет считаться допустимым в продакшене.

### Что нужно протестировать до прод-запуска

- Повторная обработка одного и того же Telegram-поста.
- Падение сервиса между успешной публикацией и записью статуса.
- Частичный успех: VK ok, OK fail, Threads timeout.
- Переполнение очереди при серии постов.
- Ошибки лимитов и истечение токенов.
- Некорректное или слишком длинное форматирование.
- Медиа, которое поддерживается в Telegram, но не поддерживается в одной из платформ назначения.

### Что можно отложить

- красивую web-панель вместо CLI
- сложные правила маршрутизации
- аналитику охватов и engagement
- двустороннюю синхронизацию статусов и комментариев
